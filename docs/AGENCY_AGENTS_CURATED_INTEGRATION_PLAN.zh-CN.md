# AgentCore OS × agency-agents 精选融合方案

这份文档的目标不是把 `agency-agents` 全量导入 AgentCore OS，
而是从中吸收“高质量专家角色 Prompt”这一套方法，
只收录少量真正与 AgentCore OS 互补、并且能稳定落地的角色。

原则只有三条：

- 稳定
- 准确
- 效率

这意味着：

- 不引入大量花哨角色
- 不导入难以验证输出质量的角色
- 不让角色数量压过工作流主线
- 不让产品退化成“很多 agent 的样板间”

## 总体判断

`agency-agents` 值得借鉴的不是“角色数量”，而是它把专家角色标准化成可重复调用的 Prompt 资产。

对 AgentCore OS 来说，最适合的融合方式是：

- 保留 AgentCore OS 现有主线：行业 -> 角色 -> 工作流 -> 审批 -> 资产
- 在工作流的关键阶段挂接少量“专家角色”
- 让专家角色服务工作流，而不是反过来主导产品结构

## 融合原则

### 1. 只选和当前系统主线互补的角色

优先服务这些已存在或正在强化的链路：

- Sales Desk
- Support Desk
- Research Desk
- Creator / SEO / Content 工作流

### 2. 只选结果可验证的角色

优先选择输出可以被人审、可以被工作流消费、可以沉淀为资产的角色。

例如：

- 线索判断
- 英文跟进邮件草稿
- FAQ 沉淀
- 研究摘要
- SEO 结构化方案
- 现实检验与风险挑刺

### 3. 每个角色必须有明确边界

每个角色都必须定义：

- 它在什么阶段被调用
- 它的输入是什么
- 它的输出是什么
- 谁来审核
- 成功标准是什么

如果这些边界不清楚，这个角色不应进入系统。

## 推荐的首批白名单角色

首批建议只保留 `7` 个角色。

这个数量已经足够体现“数字员工”能力，
同时又不会让系统失控。

### 1. Sales Qualification Specialist

定位：
负责销售 Hero Workflow 的线索资格判断。

挂载位置：

- 行业：销售与商务
- 角色：Sales Desk
- 工作流阶段：`qualify`

核心任务：

- 识别询盘质量
- 提取关键信息缺口
- 判断是否值得推进
- 输出下一步跟进建议

输出物：

- 线索资格判断摘要
- 缺失信息列表
- 推荐跟进策略
- 风险标签

为什么值得引入：

- 直接补强当前 Sales Hero Workflow
- 输出结构化、容易验证
- 能直接写回 CRM 和 sales assets

### 2. Outreach Draft Specialist

定位：
负责英文或跨语种销售跟进邮件的高质量草拟。

挂载位置：

- 行业：销售与商务
- 角色：Sales Desk
- 工作流阶段：`outreach`

核心任务：

- 基于线索判断生成跟进邮件
- 控制语气、CTA、边界与清晰度
- 根据客户语言和上下文调整表达

输出物：

- 跟进邮件草稿
- 标题建议
- CTA 建议
- 需人工确认的敏感点

为什么值得引入：

- 这是当前系统已经存在的高价值场景
- 直接提升成交推进效率
- 很适合走“AI 起草 + 人工审核”模式

### 3. CRM Follow-up Coordinator

定位：
负责把销售或商务动作同步成结构化后续推进记录。

挂载位置：

- 行业：销售与商务
- 角色：Sales Desk / Founder Desk
- 工作流阶段：`meeting` 或成交后整理阶段

核心任务：

- 生成 CRM 更新摘要
- 写出下一个动作
- 整理客户偏好和约束
- 维护后续跟进节奏

输出物：

- CRM 更新内容
- 下一步动作建议
- 触达历史摘要
- 客户偏好结构化字段

为什么值得引入：

- 有利于把一次性沟通转成长期资产
- 输出稳定、结构清晰
- 直接服务“资产沉淀”和“业务闭环”

### 4. Support Reply Specialist

定位：
负责客服/支持工作流中的建议回复生成。

挂载位置：

- 行业：客服与用户运营
- 角色：Support Desk
- 工作流阶段：`reply`

核心任务：

- 理解客户问题
- 生成清晰、礼貌、边界明确的建议回复
- 标记升级处理的情况

输出物：

- 建议回复
- 风险说明
- 是否建议升级
- 需要人工确认的点

为什么值得引入：

- 客服是高频、可标准化、非常适合数字员工的场景
- 输出质量容易做人工审核
- 能沉淀 FAQ 和标准回复资产

### 5. Knowledge Asset Editor

定位：
负责把已完成的流程结果整理成知识资产、FAQ、模板和标准规则。

挂载位置：

- 行业：销售 / 客服 / 研究 / 内容
- 角色：跨角色公共角色
- 工作流阶段：`assetize`

核心任务：

- 提炼可复用知识
- 整理模板和标准说法
- 标记适用场景和使用边界

输出物：

- FAQ 条目
- 话术模板
- 研究框架
- 可复用知识卡片

为什么值得引入：

- 直接强化 AgentCore OS 最重要的产品优势之一：资产沉淀
- 是所有工作流共用的高价值角色
- 输出结果可以被长期复用

### 6. Research Synthesizer

定位：
负责研究工作流中的结构化摘要与观点压缩。

挂载位置：

- 行业：研究与策略
- 角色：Research Desk / Founder Desk
- 工作流阶段：`synthesize`

核心任务：

- 将多源输入压成可决策摘要
- 提取观点、风险、机会
- 给出后续研究或业务动作建议

输出物：

- 研究摘要
- 核心观点
- 风险与机会
- 后续行动建议

为什么值得引入：

- 研究工作流本来就依赖“把资料堆变成判断”
- 非常适合做阶段型专家
- 易于写回知识资产层

### 7. Reality Checker

定位：
跨工作流的现实检验员，用来挑刺、找漏洞、识别过度乐观或幻觉风险。

挂载位置：

- 行业：全行业
- 角色：跨角色公共角色
- 工作流阶段：关键 review 节点

核心任务：

- 检查事实跳跃
- 检查遗漏和风险
- 检查逻辑不完整之处
- 提醒不该自动执行的地方

输出物：

- 风险清单
- 逻辑漏洞清单
- 需要人工确认的问题
- 不建议自动执行的理由

为什么值得引入：

- 这是提升稳定性和准确性的关键角色
- 能显著降低“看起来像对，其实不可靠”的问题
- 非常适合作为审批前的守门员

## 暂不引入的角色类型

这些角色即使有趣，也不建议现在进入 AgentCore OS：

- 纯娱乐型角色
- 语气花样型角色
- 社区运营 / Reddit 发帖类角色
- 过于泛化的“万能工程师”
- 难以验证结果是否正确的创意型角色
- 与现有工作流没有明确挂载点的角色

原因很简单：

- 它们不直接增强当前主工作流
- 容易增加系统复杂度
- 会稀释“稳定、准确、效率”原则

## 推荐分阶段接入顺序

### Phase 1：先接 4 个

第一阶段建议只接：

1. Sales Qualification Specialist
2. Outreach Draft Specialist
3. Support Reply Specialist
4. Reality Checker

目标：

- 先把销售与客服的高频闭环做稳
- 让 AI 输出先进入“高价值但可人工审核”的场景

### Phase 2：再接 2 个

第二阶段建议加入：

5. Knowledge Asset Editor
6. Research Synthesizer

目标：

- 强化资产沉淀
- 让研究工作流从资料堆走向结论和复用

### Phase 3：最后接 1 个

第三阶段建议加入：

7. CRM Follow-up Coordinator

目标：

- 让销售闭环的资产化和后续节奏更稳定
- 进一步打通业务动作到资产层的写回

## 每个角色的标准 Prompt 结构

建议 AgentCore OS 对每个角色使用统一结构：

1. Identity
2. Mission
3. Workflow
4. Deliverables
5. Success Metrics
6. Guardrails
7. Hand-off Rules

其中最关键的是后 3 项：

- Success Metrics：保证质量
- Guardrails：保证稳定与边界
- Hand-off Rules：保证工作流协同

## 与 AgentCore OS 的结合方式

不要把这些角色做成独立 app。

推荐做法：

- 角色作为 workflow stage 的“专家模式”
- 角色作为审批前的“复核器”
- 角色作为资产沉淀阶段的“整理器”

界面上可以表现为：

- 当前阶段使用的专家角色
- 专家角色的任务边界
- 输出置信度或风险提示
- 是否需要人工确认
- 输出写回到哪里

## 成功标准

如果融合是成功的，应该看到这些结果：

- 用户不需要自己写长提示词来定义专家
- 关键工作流输出更稳定
- 审批点更清楚
- 资产沉淀更强
- 幻觉和泛化回答明显减少
- 系统没有因为角色太多而变乱

## 最终结论

AgentCore OS 适合吸收 `agency-agents` 的“专家角色资产化”思路，
但只应该引入少量与当前工作流强相关、输出可验证、结果可沉淀的角色。

正确方向不是：

- 导入几十上百个 agent

正确方向是：

- 先做少量高价值数字员工
- 把它们牢牢挂在真实工作流节点上
- 用稳定、准确、效率三条标准持续筛选
