# AgentCore OS 公开发布说明

这份文档用于说明：

- 哪些内容适合进入公开仓库
- 哪些内容不应该进入公开发布边界
- 当前公开版本应该如何被对外描述

## 文档目的

AgentCore OS 的公开发布，不只是把代码推上去，更重要的是把产品边界讲清楚。

我们希望做到：

- 对外呈现清晰的 AgentCore OS 产品边界
- 避免暴露历史内部命名和私有操作习惯
- 把本地私有数据、运行时状态和临时文件留在仓库外
- 在必要时保留兼容层，避免已有本地环境立即断裂

## 当前公开版本口径

当前推荐公开版本为：`v1.2.0`

对外建议这样理解：

1. `v1.2.0` 是当前推荐给新用户的稳定版本
2. `v1.0.0` 是第一个稳定公开版本，可作为历史基线参考
3. 更早的 beta / alpha 标签保留为演进记录，不再作为默认入口

这意味着：

- README、安装说明、文档入口、发布说明都应优先围绕 `v1.2.0`
- 国内镜像仓库也应与此口径保持一致
- 如果需要双仓分发，中文入口优先给国内用户

## 当前版本边界

对于当前公开版本，最重要的边界有三条。

### 1. 产品边界

对外强调：

- 本地优先
- 面向真实工作流
- BYOK / API Key 驱动
- 销售、客服、研究、创作四类工作流的推荐与状态闭环

不建议对外强调：

- 已经是完整无人公司
- 已覆盖所有行业全部流程
- 已完全不需要人工审核

### 2. 仓库边界

不应进入公开仓库或发布包的内容包括：

- `.openclaw/`
- `.openclaw-data/`
- 本地运行日志
- 机器专属配置
- 私有密钥
- 临时调试文件
- 私人工作区记忆文件

应通过以下方式保证边界：

- `.gitignore`
- 发布前检查清单
- 人工复核

### 3. 兼容边界

虽然产品对外名称已经统一为 AgentCore OS，但当前仍保留部分兼容层，例如：

- `.openclaw-data` 的本地兼容
- 旧有 `openclaw.*` 本地状态的兼容迁移
- 部分旧环境变量和请求头兼容

目标不是一次性彻底切断历史，而是在不影响现有用户的前提下逐步清理公开边界。

## 发布前建议

发布前至少确认：

- README 首页是否中文友好
- 版本号是否统一为当前版本
- 发布文案是否没有承诺安装包
- 是否不存在私人信息、测试日志、临时截图、真实密钥
- `npm run lint` 与 `npm run build` 是否通过
